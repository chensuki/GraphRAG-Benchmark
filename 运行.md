# GraphRAG-Benchmark 运行指南

## 框架概览

| 框架 | 状态 | 支持数据集 | 特点 |
|------|------|-----------|------|
| **LightRAG** | ✅ 可用 | medical, novel, medical_100, hotpotqa | 需修改源码支持context返回 |
| **ClearRAG** | ✅ 可用 | medical, novel, medical_100, hotpotqa | 项目内置框架，支持精细化检索控制 |

| **HippoRAG** | ⚠️ 需适配 | medical, novel | 需添加BGE Embedding支持 |

## 环境变量设置

在使用任何框架之前，请先设置环境变量：

### 通用环境变量
```powershell
# LLM API（DeepSeek）
$env:LLM_MODEL="deepseek-chat"
$env:LLM_BASE_URL="https://api.deepseek.com/v1"
$env:LLM_API_KEY="sk-your-api-key"

# Embedding API（智谱）
$env:EMBED_MODEL="BAAI/bge-large-en-v1.5"
$env:EMBED_BASE_URL="https://open.bigmodel.cn/api/paas/v4"
$env:EMBED_API_KEY="your-zhipu-key"

# 智谱备用（兼容旧框架）
$env:ZHIPUAI_API_KEY="your-zhipu-key"
```

### ClearRAG 额外需要的环境变量
```powershell
# Neo4j 数据库连接
$env:NEO4J_URI="bolt://localhost:7687"
$env:NEO4J_USER="neo4j"
$env:NEO4J_PASSWORD="your-neo4j-password"
$env:NEO4J_DATABASE="neo4j"  # 可选，默认 neo4j
```

python .\Examples\run_lightrag.py `

#### Novel 数据集常见问题

**问题现象**：查询结果全部返回 `[no-context]`，`context` 字段为空数组 `[]`

**原因**：
- Novel数据集有20个独立语料，默认会**并发启动20个索引任务**
- 瞬间大量API请求触发限流/超时
- 索引过程被中断，文档状态停留在 `pending`，图数据未生成

**解决方案**：
1. 使用 `--sample 1` 限制处理的语料数量
2. 检查 `lightrag_workspace/Novel-XXXXX/` 目录是否有完整的图数据文件
3. 如果缺少 `graph_chunk_entity_relation.graphml`，需要重新运行索引（不使用 `--skip-build`）

## 参数说明

### 共同参数
- `--subset`: 选择数据集 (medical, novel, medical_100, hotpotqa)
- `--sample N`: 仅处理N个样本（用于快速测试）
- `--config_path`: ClearRAG配置文件路径

### LightRAG 特有参数
- `--mode`: 运行模式 (API)
- `--model_name`: LLM模型名称
- `--llm_base_url`: LLM API地址
- `--embed_provider`: 嵌入模型提供商
- `--embed_model`: 嵌入模型名称
- `--retrieve_topk`: 检索返回的文档数量
- `--base_dir`: 工作目录
- `--skip-build`: 跳过索引阶段

### ClearRAG 特有参数
- `--base_dir`: 工作目录（默认: ./clearrag_workspace）
- `--top_k`: 查询时返回的上下文数量（默认: 5）
- `--activation-mode`: 实体激活模式
  - `vector_search`（快速，无需矩阵文件）
  - `semantic_propagation`（高精度，需要矩阵文件，默认）
- `--passage-retrieval-mode`: 段落检索模式
  - `pagerank`（高精度，默认）
  - `simple`（快速）
- `--fast`: 快速模式（禁用 Reflexion）
- `--max-concurrency`: 最大并发度（1-10，默认5）
- `--corpus-concurrency`: 同时处理的语料库数量（1-3，默认2）
- `--skip-build`: 跳过构建阶段，直接进入查询阶段
- `--output_dir`: 输出目录（默认: ./results/clearrag）

### HippoRAG 特有参数
- `--mode`: 运行模式 (API 或 ollama)
- `--model_name`: LLM模型名称
- `--embed_model_path`: 嵌入模型路径或名称
- `--base_dir`: HippoRAG工作目录
- `--llm_base_url`: LLM API地址或Ollama地址
- `--llm_api_key`: LLM API密钥（可选，也可使用OPENAI_API_KEY环境变量）

### Fast-GraphRAG 特有参数
- `--subset`: 数据集选择 (`medical` 或 `novel`)
- `--mode`: LLM运行模式 (`API` 或 `ollama`)
- `--model_name`: LLM模型名称
- `--llm_base_url`: LLM API地址
- `--llm_api_key`: LLM API密钥（也可用环境变量 `LLM_API_KEY`）
- `--embed_provider`: 嵌入服务类型 (`api` 或 `local`)
  - `api`: 使用 OpenAI 兼容 API（智谱、DeepSeek 等）
  - `local`: 使用本地 HuggingFace 模型
- `--embed_model`: 嵌入模型名称（API模式）或路径（local模式）
- `--embed_base_url`: 嵌入 API 地址（默认：智谱）
- `--embed_api_key`: 嵌入 API 密钥（默认：使用 `ZHIPUAI_API_KEY` 环境变量）
- `--base_dir`: Fast-GraphRAG工作目录
- `--sample`: 限制处理的样本数量（用于测试）

### ClearRAG 检索模式说明

ClearRAG 框架支持两种实体激活模式和两种段落检索模式：

#### 实体激活模式
- **vector_search**（向量搜索）：
  - 纯向量搜索激活实体
  - 速度快，无需 Linear 索引文件
  - 适合快速测试和原型验证

- **semantic_propagation**（语义传播）：
  - 基于句子-实体矩阵的语义传播算法
  - 精度高，需要构建时导出的 Linear 索引文件
  - 适合生产环境和精确检索

#### 段落检索模式
- **pagerank**（PageRank）：
  - 使用 Personalized PageRank 算法排序段落
  - 精度高，考虑实体权重和图结构
  - 适合复杂推理任务

- **simple**（简单排序）：
  - 基于实体计数排序
  - 速度快，适合简单问答

#### 使用建议
- **快速测试**：`--activation-mode vector_search --passage-retrieval-mode simple`
- **生产环境**：`--activation-mode semantic_propagation --passage-retrieval-mode pagerank`
- **API 限流场景**：使用 `--fast` 禁用 Reflexion 自反思机制

## 评估

### 评估说明

> **重要**：评估时使用的嵌入模型不需要与检索时一致。检索评估只使用LLM判断上下文相关性，生成评估使用嵌入模型计算答案语义相似度。所有框架的评估使用相同的嵌入模型（BGE）以确保公平性。

### 检索评估（使用LLM评估上下文相关性）
```powershell
python -m Evaluation.retrieval_eval `
  --mode API `
  --model deepseek-chat `
  --base_url https://api.deepseek.com/v1 `
  --data_file .\results\clearrag\predictions_Medical.json `
  --output_file .\results\retrieval_evaluation.json `
  --num_samples 10
```

### 生成评估（使用嵌入模型计算语义相似度）
```powershell
python -m Evaluation.generation_eval `
  --mode API `
  --model gpt-4o-mini `
  --base_url https://api.openai.com/v1 `
  --embedding_model BAAI/bge-large-en-v1.5 `
  --data_file .\results\lightrag\Medical\predictions_Medical.json `
  --output_file .\results\generation_evaluation.json
```

### 索引评估
```powershell
python -m Evaluation.indexing_eval `
  --framework lightrag `
  --base_path .\lightrag_workspace `
  --folder_name Medical `
  --output .\results\indexing_metrics.txt
```

## 常见问题

### 1. 缺少依赖
如果遇到 "ModuleNotFoundError" 错误，请安装相应依赖：
```powershell
# 安装项目基础依赖
pip install -r requirements.txt

# 安装 ClearRAG
pip install -e clearrag/

# 安装 LightRAG（独立环境）
cd LightRAG && pip install -e . && cd ..
```

### 2. Neo4j 连接问题（ClearRAG）
确保 Neo4j 服务正在运行：
```powershell
# 检查 Neo4j 是否运行
docker ps | grep neo4j

# 启动 Neo4j（Docker）
docker start neo4j-clearrag

# 或使用本地安装
neo4j start
```

检查环境变量是否正确设置：
```powershell
echo $env:NEO4J_URI
echo $env:NEO4J_USER
echo $env:NEO4J_PASSWORD
```

### 3. Linear 索引文件未找到（ClearRAG）
**错误现象**：`未找到句子文本文件: ./data/linear/xxx/sentence_texts.json`

**原因**：使用了 `semantic_propagation` 模式但 Linear 索引未生成或路径不匹配

**解决方案**：
1. 确认构建时没有错误（检查日志）
2. 确认工作目录下存在 `data/linear/{database_name}/` 目录
3. 或使用 `--activation-mode vector_search` 跳过 Linear 索引

### 4. 查询结果返回 `[no-context]`
**原因**：
- 索引未完成（API 超时/限流）
- 使用了 `--skip-build` 但语料库未实际构建

**解决方案**：
1. 检查工作目录中是否有完整的图数据文件
2. 重新运行索引（不使用 `--skip-build`）
3. 降低 `--corpus-concurrency` 参数（建议 1-2）

### 5. API 限速问题
如果遇到 API 限速错误，可以：
- 降低并发数：`--max-concurrency 3 --corpus-concurrency 1`
- 使用快速模式：`--fast`（禁用 Reflexion）
- 使用向量搜索模式：`--activation-mode vector_search`
- 减少样本数：`--sample 5`

### 6. 内存不足
如果遇到内存问题，可以：
- 减少 `--max-concurrency` 参数（默认 5，可降至 1-2）
- 使用更小的 `--sample` 值
- 使用 `--skip-build` 跳过构建阶段

### 7. 为什么评估使用不同的嵌入模型？
评估时使用的嵌入模型（BGE）与检索时使用的嵌入模型（如智谱 embedding-3）可以不同，原因如下：
- **检索评估**：只使用 LLM 判断上下文相关性，不使用嵌入模型
- **生成评估**：使用嵌入模型计算答案与标准答案的语义相似度，这是独立的评估任务
- **公平性保证**：所有框架的答案都用同一个嵌入模型评估，确保公平性
